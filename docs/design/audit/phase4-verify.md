## Architectural notes

### Focus-ring migration

The sed sweep replaces `ring-ring` (which Tailwind resolves to `--ring` =
`--primary`) with `ring-focus-ring` (which resolves to the dedicated
`--focus-ring` token introduced in Phase 1). All 25 affected files now
consume the dedicated focus token, so future a11y tuning of focus
contrast won't ripple into primary-button color or chart series.

Today `--focus-ring` and `--primary` happen to share a value — but they
are now structurally separable. Phase 4's win is **making them
separable**, not making them different.

### Global reduced-motion

```css
@media (prefers-reduced-motion: reduce) {
  *, *::before, *::after {
    animation-duration: 0.01ms !important;
    animation-iteration-count: 1 !important;
    transition-duration: 0.01ms !important;
    scroll-behavior: auto !important;
  }
}
```

Universal selector, four properties, `!important`. Per-component
`motion-reduce:` Tailwind classes still work and remain useful when a
specific component needs nuanced behavior (e.g., a checkmark that
should fade in even at reduced motion). The global rule covers the
default and removes a class of "I forgot to add motion-reduce" bugs.

### Input `aria-invalid` intent

```tsx
"aria-[invalid=true]:border-destructive aria-[invalid=true]:focus-visible:ring-destructive"
```

Tailwind's arbitrary-attribute selectors. Consumers pass `aria-invalid={true}`
or `aria-invalid="true"` — the input gets a destructive border and a
destructive-tinted focus ring. No new prop; no breaking change for the 23
existing call sites. The `login/page.tsx` form currently computes
`className="... ${touched && error ? 'border-destructive' : ''}"` — that
pattern can now be replaced with `aria-invalid={touched && !!error}`
opportunistically.

### Contrast-matrix-driven token corrections

The matrix surfaced two genuinely-broken light-mode pairs that the live
axe scan didn't catch (because there were no rendered consumers yet —
Badge `success`/`warning` shipped in Phase 3 but no live `<Badge variant="warning">` exists yet):

- `text-warning-foreground` (`white`) on `bg-warning` (amber): **2.14:1**
  — fails everything.
- `text-success-foreground` (`white`) on `bg-success` (mid-green): 3.35:1
  — fails AA-normal.

Fix: light-theme `*-foreground` tokens for `success` and `warning`
swapped from white to near-black (`0 0% 4%`). Dark theme already used
near-black. The light/dark consistency is a real improvement — and
white on amber was a latent bug that would have shipped the moment
someone used `<Badge variant="warning">`.
