The remaining gap implied by the PRD body is the **HITL companion #573** (configure the experiment in PostHog itself — flag, metrics, runbook) — the code-side decider (`decideWhatYouLearnedCard`) exists, but the experiment configuration is a console task, not an agent task. **Implementation reference (holdout pattern, applicable to both #570 and any future "is this card worth keeping" follow-up):** PostHog's native **holdout group** mechanism — a long-running slice of users permanently excluded from a feature once it ships, used to measure cumulative/long-term impact — is the standard complement to a launch A/B. If #570's go/no-go is "keep-and-invest vs. remove," a holdout (rather than a second short A/B) is the recommended way to validate the decision over weeks without re-instrumenting.

### 1.2 Issue #555 — Activation drop-off cohorts: entry_path + authoring_mode + time_to_activation (ready-for-agent, ADR-0009)

This PRD splits the legacy `entry_path: manual` value into `blank_canvas | template_clone`, adds an orthogonal `authoring_mode: nl | manual`, and persists `entry_path` on the Strategy model so it survives reloads. Two implementation notes from today's research:

- **PostHog funnel time-to-convert** is computed natively from step-to-step event timestamps in a funnel — "Time to activation" (`signup_completed → results_viewed`) does not need a stored duration field; it's a PostHog-side funnel property, consistent with ADR-0009's framing as a "diagnostic, never the headline" metric (same stance ADR-0014 took for the onboarding A/B's secondary metric).
- Persisting `entry_path` on the Strategy row (point 2 of the PRD) is the only schema change in this slice — straightforward additive column + migration, matching the "MANY SMALL FILES" / additive-migration patterns already used for prior FEAT work (e.g., `data_quality_metrics.earliest_candle_date`/`latest_candle_date` in product.md §6.4).

### 1.3 Issue #620 — PostHog `onboarding_ab` experiment config (ready-for-human, HITL)

Still blocked on PostHog console access (unchanged from yesterday). One new consideration from today's identify/exposure research, relevant to the SRM guardrail ADR-0014 specifies:

- PostHog's JS SDK updates `distinct_id` **synchronously** on `identify()`, but if a `capture()`/`getFeatureFlag()` call was already in flight in the same execution frame, that call can still resolve against the **anonymous** ID. For the `auth/callback` routing fork (where #619 just shipped the `resolveOnboardingArm` read), the practical guidance is: call `identify(user.id)` **before** the `getFeatureFlag('onboarding_ab', ...)` read that drives routing, not after — otherwise the exposure event and the eventual person-merge can attribute the assignment to two different distinct_ids, which is exactly the kind of skew the SRM check (#620) is designed to catch. Worth a quick code check on #622's merged implementation before configuring the experiment.
