## Security Risks

### RESOLVED — Redis server CVE-2026-23631 / CVE-2026-23479

Tracked as CRITICAL/day-6-unpatched on 2026-06-27. **Confirmed resolved on inspection of the actual repo state**: `docker-compose.yml` and `docker-compose.prod.yml` both pin `redis:7.4.9-alpine`, which Redis's own security advisory lists as a fixed release for CVE-2026-23631 (DarkReplica, post-auth RCE) and CVE-2026-23479 (UAF RCE), alongside CVE-2026-25243/25588/25589 from the same disclosure batch. Source: [Redis security advisory](https://redis.io/blog/security-advisory-cve202623479-cve202625243-cve-2026-25588-cve202625589-cve-2026-23631/).

The prior escalation appears to have conflated the Docker server image with `backend/requirements.txt`'s `redis==7.1.1`, which is the **Python client library** (`redis-py`) — an unrelated artifact with no CVE history found in any search this cycle. No action needed. If a future cycle re-flags "Redis 7.1.1," check whether it means the client or the server before escalating — they are pinned independently and currently at different versions by design.

### LOW — Starlette unbounded version floor (CVE-2026-48710, "BadHost")

`backend/requirements.txt` pins `starlette>=0.49.1` with no ceiling, and there is no lockfile (`requirements.txt` is the only pin in `backend/`). CVE-2026-48710 is a Host-header authentication bypass, fixed in Starlette 1.0.1 (2026-05-21), CVSS 6.5. The vulnerable range (`>=0.49.1, <1.0.1`) overlaps the floor.

**Why this is LOW, not HIGH:** with no lockfile, every fresh `pip install -r requirements.txt` resolves to the latest compatible Starlette — currently 1.3.1, already patched. This is not exploitable in a build performed today. The risk is reproducibility: a Docker layer cached before 2026-05-21, or any future scenario where pip's resolver picks an older compatible version (e.g. a FastAPI downgrade that caps Starlette lower), would silently reintroduce the bug with no visible signal.

**Action:** Change `starlette>=0.49.1` to `starlette>=1.0.1` in `backend/requirements.txt`. One-line, no behavior change, closes the gap permanently regardless of build-cache state.

## Implementation References

### Block Catalogue dispatch-table pattern (#509)

The fix is a standard dictionary-dispatch refactor: replace the `if block_type == "sma": ... elif block_type == "ema": ...` ladders in `explanation.py` (three separate ladders) and `explanation-generator.ts` (a fifth, TypeScript-side fork) with lookups into the same catalogue keyed by `block_type` that `interpreter.py:104`'s `catalogue_lookup(block_type)` already uses. References: [Dispatch Tables in Python](https://medium.com/better-programming/dispatch-tables-in-python-d37bcc443b0b), [Dictionary Dispatch Pattern](https://martinheinz.dev/blog/90).

### Pydantic → Zod codegen for shared validation (#510)

Two candidate tools for generating `frontend/src/components/canvas/ParameterForm.tsx`'s hand-typed validation from `backend/app/services/strategy_validation.py`:
1. **[pydantic2zod](https://github.com/argyle-engineering/pydantic2zod)** — direct Pydantic→Zod, preserves names; small project (~65 stars), needs a spike against the take-profit-ladder custom validators before relying on it.
2. **JSON Schema intermediary** — `model_json_schema()` → `json-schema-to-zod`, more tooling-agnostic if custom validators don't survive direct conversion (see [datamodel-code-generator discussion on custom validators](https://github.com/koxudaxi/datamodel-code-generator/discussions/2212)).

Either path should wire into CI as a drift check (fail the build if generated Zod is stale).

### Atomic submit() / commit-then-enqueue (#511)

This is the transactional-outbox problem applied to PostgreSQL + Redis/RQ: commit the run row, then enqueue, with the queue operation never able to leave a `PENDING` run as the only state if it throws. Direct prior art: `django-rq` v3.0 changed its default enqueue behavior to "only after the DB transaction commits" to fix this exact bug class — the `submit()` spec for #511 should replicate that semantic on raw RQ (commit first via an on-commit hook, enqueue after, leave a retryable row on enqueue failure rather than an orphan). References: [Transactional Outbox Pattern](https://thecodeforge.io/system-design/transactional-outbox/), [django-rq on_db_commit](https://pypi.org/project/django-rq/).
