## Findings for Active Issues

### Issue #618 тАФ NL-wedge onboarding A/B harness (`ready-for-agent`, `[P1 ┬╖ S]`)

**What it needs:** A pure decision function `resolveOnboardingArm`, a thin enrollment hook `useOnboardingArm`, and routing-fork wiring in the OAuth/post-auth callback. Mirrors the `wjl_retention_ab` pattern exactly.

**Implementation references:**
- Prior art to clone: `frontend/src/lib/__tests__/experiment-variant.test.ts` (pure truth-table tests) and `frontend/src/hooks/__tests__/useWjlCardEnrollment.test.ts` (hook with mocked variant).
- PostHog flag key to add: `onboarding_ab` (variant strings `"control" | "test"`), layered on top of the existing `strategy_drafter_enabled` boolean kill-switch.
- Routing fork: the OAuth callback currently hard-codes the push to `?wizard=true` for un-onboarded users; replace with the route returned by the hook.
- The NL arm routes to the existing NL box (ADR-0011/0012/0013); no new surface needed. A thin query param (`?onboarding_arm=nl_wedge`) may be needed as an entry signal.
- Exposure event: `onboarding_ab_enrolled` carrying `arm` (`wizard | nl_wedge`), fired once via the consent-gated `trackEvent`, keyed to `user.id`.
- **ADR-0014** is the design record; the backlog wording ("NL box vs. blank canvas") is stale тАФ control is the wizard.

**PostHog setup (HITL prerequisite тАФ issue #620):** The `onboarding_ab` flag and PostHog experiment must be configured in PostHog before the frontend code is useful. ADR-0014 specifies 50/50, two-sided 95% / 80% power on activation rate. HITL issue #620 covers this setup.

**Tests:** Unit-test the pure decider truth table (4 cases: `undefinedтЖТwizard/enroll:false`, `controlтЖТwizard/enroll:true`, `testтЖТnl_wedge/enroll:true`, `hasCompletedOnboarding:trueтЖТno enrollment`). Hook test: variant read once, exposure event fired exactly once, not fired when `enroll:false`.

**Risk:** `risk-medium` тАФ modifies the OAuth callback routing for new users. Existing onboarded users are untouched. The pure-function architecture makes this safe to unit-test before touching routing.

### Issue #570 тАФ "What you just learned" retention A/B (`ready-for-agent`, `[P0 ┬╖ S]`)

**What it needs:** A variant-aware `getExperimentVariant` extension returning `"control" | "test" | undefined` (the existing boolean `getFeatureFlag` is insufficient for this use case), a pure `decideWhatYouLearnedCard` function, and thin wiring at the first-run results render site.

**Key naming clarification from the issue:** There is no `showFirstRunExplanations` flag in code. The real gate is the `bb.first_run_summary_card_seen` localStorage key. The always-on backend `NarrativeCard` is separate and must remain on in both arms.

**Implementation references:**
- PostHog flag key: `wjl_retention_ab` (variants `"control" | "test"`).
- The pure function signature: `decideWhatYouLearnedCard({ variant, hasBenchmarkData, alreadySeen }) тЖТ { showCard, closeGate }`.
  - `test` + not-seen + benchmark present тЖТ show card, close gate immediately after show.
  - `control` + benchmark present тЖТ suppress card, close gate immediately.
  - `undefined` (no consent) тЖТ show card, gate untouched (current behavior).
  - No benchmark data тЖТ suppress card, never enroll regardless of variant.
- Flag read location: the results page, at the first-run conditional, read once per load.
- Prior art: the `wjl_retention_ab` ADR-0010 design document and the existing `useWjlCardEnrollment` hook pattern.

**Metrics (PostHog-side, no schema changes):** Primary: second-backtest-rate (тЙе1 `results_viewed` on a distinct second backtest within 7 days). Secondary: 7-day return-rate. Guardrail: activation rate flat across arms.

**HITL prerequisite:** Issue #573 тАФ configure the `wjl_retention_ab` experiment and metrics in PostHog.

**Tests:** Pure decider truth table (6 cases covering all variant ├Ч benchmark ├Ч alreadySeen combinations). Results page integration test: assert card renders / does not render per arm.

### Issue #555 тАФ Activation drop-off cohorts (`ready-for-agent`, `[P0 ┬╖ S]`)

**What it needs:** A new `entry_path` column on the `strategies` table (Alembic migration), a pure cohort derivation function, and updated event instrumentation on `strategy_created`, `backtest_started`, and `results_viewed`.

**Schema change:**
- Add `entry_path` (VARCHAR, nullable) to `strategies` table. Values: `wizard | blank_canvas | template_clone | nl_wedge`. Existing rows backfill as `unknown` (explicit cohort for pre-instrumentation data).
- `authoring_mode` is *derived* from `entry_path`, not stored: `nl_wedge тЖТ "nl"`, all others тЖТ `"manual"`. This keeps the schema minimal (YAGNI).

**Pure cohort function:**
```typescript
function deriveActivationCohort(entryPath: string | null): 
  { entry_path: string; authoring_mode: 'nl' | 'manual' } {
  // entry_path null тЖТ 'unknown'
  // authoring_mode: nl_wedge тЖТ 'nl', else 'manual'
}
```

**Event changes:**
- `strategy_created`: add `entry_path` + `authoring_mode` properties. Template clone path currently emits no milestone тАФ fix this.
- `backtest_started` / `auto_backtest_started`: read `entry_path` from the *persisted strategy* (not from component state), carry both properties.
- `results_viewed`: same тАФ read from persisted strategy, not re-derive from routing.
- The `manual` legacy value is retired; blank-canvas тЖТ `blank_canvas`, template тЖТ `template_clone`.

**HITL prerequisite:** Issue #561 тАФ build the 4-step diagnostic funnel in PostHog with attribution at the "strategy authored" step.

**Migration note:** This requires migration `030_add_entry_path_to_strategies.py`. Risk is low тАФ additive-only column, nullable with a safe default.

### Issue #334 тАФ PostgreSQL 18 evaluation (`needs-plan`, `needs-human`, `risk-high`)

PostgreSQL 18 was released in 2025/2026 with three features directly relevant to Blockbuilders' backtest workloads:

1. **Async I/O (AIO) subsystem:** Concurrent I/O requests instead of sequential. Candle range scans and equity curve S3 reads are the natural beneficiaries.
2. **Skip scan on multicolumn B-tree indexes:** Directly benefits `candles(asset, timeframe, timestamp)` queries where `asset` is fixed but `timeframe` is omitted or repeated.
3. **Virtual generated columns:** Useful for derived fields (e.g., a computed `candle_date` from `timestamp`) without storage overhead.

PostgreSQL 18.4 (latest patch) is the safe upgrade target. Existing `postgres:15` Docker image in `docker-compose.yml` would need updating. Requires `pg_upgrade` or dump/restore. **Recommend advancing #334 from `needs-plan` to a spec/evaluation phase.**

## Security Risks

### HIGH тАФ Redis CVEs (May 2026): **upgrade recommended**

Project is on Redis 7.1.1 (Docker image `redis:7-alpine`). The May 2026 advisory patched five CVEs in Redis 7.2.14, 7.4.9, and later. **Redis 7.1.x is below the patched threshold.**

| CVE | Severity | Description |
|---|---|---|
| CVE-2026-23479 | High | Use-after-free in unblock client flow тЖТ potential RCE |
| CVE-2026-25243 | High | RESTORE command memory access тЖТ potential RCE |
| CVE-2026-25588 | High | RESTORE + RedisTimeSeries module |
| CVE-2026-25589 | High | RESTORE + RedisBloom module |
| CVE-2026-23631 | Medium | Lua use-after-free via replica sync |

**Mitigating factors:** Redis in this project is Docker-internal (not publicly exposed), requires authentication, and is not serving untrusted clients. The RESTORE command is not used by application code. Risk is **reduced but not eliminated** тАФ a compromised API or worker container could pivot.

**Recommended action:** Update `docker-compose.yml` `redis:7-alpine` тЖТ `redis:7.2-alpine` (or `redis:8-alpine` for Redis 8.x). This is a patch-level change; no API changes expected for RQ 2.7.0.

The project is already on Next.js 16.2.6, which is the patched version for the May 2026 13-CVE coordinated release. **No action needed.**
