## Security Risks

### HIGH — joserfc exact-pin DoS in the JWT auth path (CVE-2026-48990)

`backend/requirements.txt` pins `joserfc==1.6.5` (exact version, no range). **This is the app's actual JWT library** — `backend/app/core/security.py` uses `joserfc.jwt.encode`/`decode` for session tokens. CVE-2026-48990 (CVSS 5.3, Medium) affects joserfc 1.3.4 through 1.6.5 inclusive: RFC7797 `b64=false` JWS payloads bypass `JWSRegistry.max_payload_length` during deserialization, letting an attacker smuggle an oversized JWS token past configured size limits and trigger memory/CPU exhaustion on any endpoint that deserializes untrusted JWS — which, for a JWT auth library, is every authenticated request path. Fixed in **1.6.7** (released 2026-05-23).

**Why this ranks above the Starlette item:** unlike Starlette's unbounded-floor gap (which self-heals to a patched version on any fresh install because nothing constrains the resolved version), `joserfc==1.6.5` is an **exact pin** — it will never resolve to anything but the vulnerable version until someone edits the line. There is no accidental self-healing here.

**Action:** bump `joserfc==1.6.5` → `joserfc==1.6.7` (or later) in `backend/requirements.txt`. One-line change; check the 1.6.6/1.6.7 changelog for any breaking API changes before merging (none expected for a patch-only bump, but verify since this touches the auth path — per AGENTS.md, changes to auth require explicit human approval before merge).

Source: https://nvd.nist.gov/vuln/detail/CVE-2026-48990 · https://github.com/authlib/joserfc/releases/tag/1.6.7

### LOW — Starlette unbounded version floor (CVE-2026-48710 "BadHost", CVE-2026-54283 form-DoS)

Unchanged from the last two cycles: `starlette>=0.49.1` has no ceiling and no lockfile. Two CVEs fall in theory within the unbounded range — CVE-2026-48710 (BadHost, Host-header auth bypass, fixed 1.0.1) and, newly named this cycle, **CVE-2026-54283** (`request.form()` silently ignoring `max_fields`/`max_part_size` on urlencoded bodies, DoS, fixed in 1.3.1, disclosed 2026-06-15). Both self-heal on a fresh install: FastAPI 0.137.2+ internally requires Starlette ≥1.0.0, and the actually-resolved version today is 1.3.1 for both the FastAPI constraint and a bare `pip install`. Same recommendation as before: pin the floor to `starlette>=1.0.1` (or `>=1.3.1` to also cover CVE-2026-54283 explicitly) to close the reproducibility gap regardless of future build-cache or resolver behavior.
