## Findings for Active Issues

### Issues #618, #570, #555 — All blocked on PostHog HITL

No code changes needed. Blockers are entirely in PostHog console configuration.

**#618 (NL-wedge A/B) — HITL #620:**
PostHog's Next.js A/B test pattern requires:
1. Server-side flag bootstrap via `posthog-node` with `distinct_id` from cookies (prevents flash of wrong variant at hydration).
2. Sticky assignment via `$set_once` property on first `onFeatureFlags` callback — ensures the user never switches variant between sessions.
3. Experiment enrollment exclusion: set `has_completed_onboarding=true` users as exclusion criteria in the PostHog experiment targeting filter, not just in code, so the PostHog dashboard computes sample sizes correctly.
4. Intent-to-treat design: fire `onboarding_ab_enrolled` at flag evaluation time (not at first backtest), so the control arm is measurable even if users never reach activation.

**PostHog docs confirm:** `distinct_id` from cookies + `getAllFlags` server-side + pass to `PHProvider` bootstrap is the canonical Next.js App Router pattern. The `onFeatureFlags` callback wrapping used in `wjl_retention_ab` is valid and reusable.

**#570 (WJL retention A/B) — HITL #573:**
Same sticky-flag implementation notes apply. Key enrollment constraint: only benchmark-present, completed, first-run backtest results can enroll. This cannot be a flag-targeting rule (it's runtime state) — it must be enforced in code at the `decideWhatYouLearnedCard` call site. The PostHog experiment's reported sample size will be smaller than total users; document this in the runbook.

**#555 (Activation cohorts) — HITL #561:**
No new technical findings. The `entry_path` cohort derivation (wizard/template/blank) is pure, Alembic-additive, and zero-risk. Awaiting funnel configuration only.

### Issue #334 — PostgreSQL 18 evaluation (`needs-plan`, `needs-human`, `risk-high`)

**PostgreSQL 18.3 is the current stable release** (February 26, 2026). Key performance features confirmed relevant to this project:

| Feature | Relevance to Blockbuilders |
|---------|---------------------------|
| **Async I/O (AIO)** | Up to 3× improvement on sequential and bitmap heap scans. Directly benefits `candles` range queries (`SELECT … WHERE asset=? AND timeframe=? AND timestamp BETWEEN ?`) and equity curve reads. Enabled via `io_method` server variable; `io_combine_limit` controls batch size. |
| **Skip-scan B-tree** | The `candles(asset, timeframe, timestamp)` composite index can now be used for queries that omit an equality filter on `asset` or `timeframe` — previously required a full scan. High impact on the daily validation job and data quality checks that scan across assets. |
| **Virtual generated columns** | Compute-at-query-time derived columns on `backtest_runs` (e.g., `cost_pct_gross_return` recomputed from stored fields) without schema churn. Potential for removing Python-side calculations from the results endpoint. |
| **uuidv7() built-in** | Better-indexed primary keys for new tables. Deferred — out of scope for #334 evaluation, but document as a future migration candidate. |
| **Alembic compatibility** | No breaking changes with Alembic 1.18.5 and PostgreSQL 18.3. Transactional DDL remains intact. |

**Recommendation:** Send #334 to `/plan-feature`. The AIO improvement is directly measurable: a benchmark comparing `GET /backtests/{id}` response times on PostgreSQL 17 vs 18.3 under load would give a quantified go/no-go signal within one planning sprint.

## Security Risks

### CRITICAL — Redis CVE-2026-23631 (DarkReplica) · Day 8 unpatched

**Severity:** CVSS 8.8 (post-auth) → full host RCE  
**Affected:** Redis 7.2.0–7.2.13 (project uses 7.1.1, also affected)  
**Mechanism:** Use-after-free in Lua scripting subsystem triggered via `SLAVEOF` command during replication. Requires authenticated access (Redis default: no auth in dev Compose; production auth status unknown).  
**Fixed in:** Redis 7.2.14 (May 5, 2026), also 7.4.9, 8.x branches  
**Fix:** In `docker-compose.yml`, change `image: redis:7.1.1-alpine` → `image: redis:7.2.14-alpine`  
**Risk if deferred:** PoC exploit is public on GitHub (`yoyosh/DarkReplica`). Any authenticated Redis connection can achieve RCE → full host compromise.

**Severity:** Batch — includes SSRF, cache poisoning, XSS variants beyond the May 2026 set  
**Affected:** Next.js < 16.2.9 in self-hosted deployments

Not a CVE. The per-handle store subscription pattern in 12.10.x causes O(n) selector evaluations on every canvas state update. On complex strategies with many blocks (20+ handles), this can cause input lag in the block parameter editors. Fixed in 12.11.1.

## UX Patterns for Visual Strategy Builders

**Market context:** The no-code algo trading market is at $25B in 2026 (15.4% CAGR to $44B by 2030). NL-first entry is now table-stakes for new entrants.

### NL-to-strategy pattern (LuxAlgo "Quant")

LuxAlgo's current NL flow: describe in plain text → AI generates Pine Script v6 → preview output → refine with follow-up prompts. The iterative "prompt → generate → preview → refine" loop is the dominant pattern for 2026 NL strategy entry.

**Blockbuilders differentiation opportunity:** LuxAlgo generates code (Pine Script). Blockbuilders generates visual blocks — a more inspectable, learnable artifact. This is a structural advantage for the "learn by doing" positioning. The NL-wedge (#618) should emphasize the visual output as the differentiator in copy: "See your strategy as blocks, not code."

### Node-based UX best practices (2026 synthesis)

| Pattern | Recommendation for Blockbuilders |
|---------|----------------------------------|
| **Snap-to-grid** | Already implemented. Ensure grid density is adjustable (modifier key to override). |
| **Alignment guides** | Not yet implemented. Show guide lines when a dragged node aligns with an existing node's center or edge. Low implementation cost, high perceived quality. |
| **Minimap with section shortcuts** | Already implemented (4.11.2). Confirmed best practice — competitors (Bitsgap, n8n) all ship this. |
| **Health bar / completeness meter** | Already spec'd (4.17). Competitors show this as a required-field indicator. Consider the collapsed icon-only mode as the default for returning users. |
| **Category-based block palette with search** | Already spec'd (4.15). The "recent blocks" strip at the top is validated by n8n and Zapier — reduces friction for repetitive workflows. |
| **Auto-layout** | Already spec'd (4.16). Left→Right flow is the expected default for signal chains (input → indicator → logic → signal). |
| **Connection tapping on mobile** | Not yet implemented. Drag-to-connect is the main pain point for touch users. Tap-to-start + tap-to-end is the established mobile alternative (see spec 4.11). |

### Onboarding funnel UX (relevant to #618)

Research confirms: users who reach a meaningful result (first backtest completion) in session 1 have materially higher day-7 return rates. The NL-wedge's one-click "See how it would have performed" CTA (spec 4.4) is directly aligned with this finding. The A/B test should measure **time-to-first-backtest**, not just activation, as a secondary metric.

## Implementation References for Open AI-Ready Issues

### PostHog sticky A/B experiment in Next.js App Router

```typescript
// Pattern: server-side bootstrap + client sticky enrollment
// File: frontend/src/app/(app)/layout.tsx or middleware.ts

// 1. Server side: get flags for distinct_id
import { PostHog } from 'posthog-node'
const client = new PostHog(process.env.NEXT_PUBLIC_POSTHOG_KEY!)
const flags = await client.getAllFlags(distinctId)

// 2. Pass to PHProvider bootstrap (eliminates client-side flag fetch on mount)
<PHProvider bootstrap={{ distinctID: distinctId, featureFlags: flags }}>

// 3. Sticky enrollment: use $set_once so variant never changes after first evaluation
posthog.capture('onboarding_ab_enrolled', {
  $set_once: { onboarding_ab_variant: variant }
})

// 4. Read variant (mirrors wjl_retention_ab pattern)
const variant = posthog.getFeatureFlag('onboarding_ab')
```

**Reference:** [PostHog Next.js A/B tests tutorial](https://posthog.com/tutorials/nextjs-ab-tests) · [Sticky feature flags](https://posthog.com/tutorials/sticky-feature-flags)
