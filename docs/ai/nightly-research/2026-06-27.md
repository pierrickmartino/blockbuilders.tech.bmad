### Issue #334 — PostgreSQL 18 evaluation (`needs-plan`, `needs-human`, `risk-high`)

**Status: No change.** With ADR-0025 complete and the backend in a stable architectural state, this is the best possible moment to advance #334 to a plan. New PostgreSQL 18 information this cycle:

- **PostgreSQL 18.3** released (~June 2026); this is now the latest stable. The evaluation issue opened against 18.0 is still valid — 18.3 is the correct target.
- **Async I/O (AIO) subsystem**: Demonstrated up to 3× read performance improvement. Directly relevant to `GET /backtests/{id}/equity-curve` (reads equity curve from MinIO but queries candles table) and all `candles` range scans.
- **Skip-scan B-tree indexes**: Multi-column indexes usable in more cases. The `candles(asset, timeframe, timestamp)` composite index is the primary beneficiary.
- **Virtual generated columns**: Compute-at-query-time. Potential for derived metrics on `backtest_runs` without schema churn.
- **uuidv7() built-in**: Better indexing/read performance vs UUID v4. Long-term migration candidate for primary keys (deferred, out of scope for evaluation).
- **Alembic 1.18.5 compatibility**: No known breaking changes with PostgreSQL 18.3. Alembic's transactional DDL support remains intact.

**Recommendation:** Advance #334 to `/plan-feature` in the next cycle. The AIO and skip-scan improvements are directly measurable on this workload.

## Security Risks

### CRITICAL ESCALATION — Redis 7.1.1 — DarkReplica (CVE-2026-23631) — **Day 6**

This was HIGH on 2026-06-25. At day 6, the window is unreasonable for a fixable vulnerability with a one-line resolution.

**CVE-2026-23631 (DarkReplica, CVSS ~9.0, post-auth RCE):** An authenticated attacker instructs Redis to become a replica of an attacker-controlled master; during sync, arbitrary code executes via the incoming RDB file. Vector is authenticated, but in any environment where the Redis port is reachable by more than one container, this is exploitable.

**CVE-2026-23479 (authenticated UAF RCE):** Use-after-free in blocking-client handling. Introduced in Redis 7.2.0 — affects 7.2.0–7.2.13, 7.4.0–7.4.8, 8.x. Fixed in 7.2.14+. The project is on 7.1.1, which is in the 7.x family but **may not be directly affected by 23479** (introduced in 7.2.0). Confirm exact Redis version in `docker-compose.yml`.

| CVE | Severity | Affects | Fixed in |
|---|---|---|---|
| CVE-2026-23631 (DarkReplica) | HIGH/CRITICAL | Redis ≤7.1.x | 7.2.14+ |
| CVE-2026-23479 | HIGH | Redis 7.2.0–7.2.13 | 7.2.14 |

**Action:** Change `redis:7.1.1` → `redis:7.2.14` in `docker-compose.yml` and `docker-compose.prod.yml`. Test RQ job queue on updated image before deploy. This is a one-line change with low blast radius.

## Implementation References

### PostHog Experiment Configuration (for #618 / #570 when HITL gates clear)

Official patterns from PostHog documentation applicable to both experiments:

1. **Flag creation**: Use PostHog wizard — three steps: Description (key, hypothesis), Variant rollout (50/50, rollout %), Analytics (inclusion filter + metrics). Key insight: **filter ineligible users in application code before evaluating the flag** (e.g., exclude `has_completed_onboarding=true` users from `onboarding_ab`).

2. **Exposure event discipline**: Fire one `{experiment}_enrolled` event at the routing fork — not on every page load. Both experiments use intent-to-treat design; the exposure event defines the analyzed population.

3. **Guardrails**: Sample-ratio-mismatch check is native in PostHog experiments (enable in Guardrails tab). Add custom guardrail metric for drafter cost/error rate as a secondary metric in negative direction.

4. **Persistence**: Use `user.id` (not anonymous ID) as the assignment key. Both `wjl_retention_ab` and `onboarding_ab` must be sticky per logged-in user. PostHog's person-based flags handle this automatically when `distinctId` is the user UUID.

**Reference:** [PostHog Experiments Best Practices](https://posthog.com/docs/experiments/best-practices)

### Auto-Layout for Canvas (#509 Block Catalogue context)

When the Block Catalogue refactor lands, the canvas auto-layout feature (product.md §4.16) will need a layout engine. Current React Flow / XyFlow options:

| Engine | Best for | Notes |
|---|---|---|
| Dagre | DAG/tree flows (left→right) | Unmaintained but stable; minimal config |
| ELK.js | Complex graphs, configurable | Java-ported to JS; most powerful |
| D3-hierarchy | Tree-only | Simplest for pure trees |

For Blockbuilders' strategy graph (DAG with entry → indicator → logic → signal → exit flow), **Dagre is the pragmatic choice**: it handles directed acyclic graphs, requires no new dependencies beyond what's already in the ecosystem, and produces predictable left-to-right layouts. ELK.js would add ~500KB gzipped and is overkill for graphs under 30 nodes. The `useAutoLayout` hook pattern from the React Flow playground can be adopted directly with minimal modification.

**Reference:** [React Flow Auto Layout example](https://reactflow.dev/examples/layout/auto-layout)
