### 1.2 Issue #619 — Onboarding A/B harness, code spine (`ready-for-agent`, parent #618, ADR-0014)

Scope: at the OAuth-callback routing fork (currently a hard-coded push to the wizard), read the `onboarding_ab` flag (`"control" | "test" | undefined`) via the existing `getExperimentVariant`, and route un-onboarded users to the **wizard** (control) or the **NL box** (`nl_wedge`, treatment). A pure `resolveOnboardingArm` decider must encode:
- already-onboarded → no experiment, `enroll: false`
- `strategy_drafter_enabled` kill-switch off → force `wizard`, `enroll: false`, and **never** silently re-route an in-flight `test` user
- otherwise → route per variant, fire `onboarding_ab_enrolled` exposure event once

This mirrors the existing ADR-0010 (`wjl_retention_ab`) pattern: pure decision function + thin enrollment hook (`useWjlCardEnrollment`-style).

**Relevant implementation references found today** (see §3 for sources):
- PostHog's `getFeatureFlag` call **auto-fires `$feature_flag_called`** with `$feature_flag`/`$feature_flag_response`, deduplicated per identity — this can serve as (or back) the `onboarding_ab_enrolled` exposure event without hand-rolled dedup logic.
- For a flicker-free server-side routing decision in the OAuth-callback route handler, use `posthog-node`'s `getFeatureFlag`/`evaluateFlags` before issuing the redirect, rather than deciding client-side post-hydration.
- `posthog-js` 1.353.0 added persisting `$user_state` to a cookie for server-side reads — potentially useful for identity continuity at the callback fork if the user was anonymous pre-OAuth and gets identified mid-flow (worth checking `identify()` ordering doesn't fragment the assigned variant).

### 1.3 Issue #620 — PostHog experiment configuration (`ready-for-human`, HITL)

Stand up `onboarding_ab` at 50/50, keyed by `user.id`, with:
- **Primary:** activation rate (≥1 `results_viewed` per enrolled user, ITT from `onboarding_ab_enrolled`)
- **Secondaries:** 7-day return-rate (verbatim ADR-0010 definition), time-to-activation (diagnostic only)
- **Guardrails:** sample-ratio-mismatch (SRM) + drafter cost/error rate cap (ADR-0011 §9)

**Reference for the SRM guardrail:** PostHog auto-computes SRM once an experiment reaches ≥100 total exposures using a chi-squared test at p < 0.001, flagging deviation from the configured 50/50 split. Common causes worth pre-checking for this experiment: bot traffic hashing deterministically into one arm, mid-experiment flag-condition edits, and identity-merge timing at the OAuth callback (anonymous→identified ID merge could skew which arm a user resolves to if `identify()` fires after the flag read).

### 1.4 Architecture-review refactors (#509, #510, #511 — all `needs-spec`/`risk-medium`, from the 2026-06-02 review)

All three describe the same shape: behavior forked across call sites that should sit behind one seam. Implementation patterns found today (sources in §3):

- **#509 — unify the 5-way block-evaluator fork** (Python interpreter dispatch via the Block Catalogue, 3 re-implementations in `explanation.py`, 1 TypeScript re-implementation in `explanation-generator.ts`). The proposed `BlockHandler.evaluate_and_describe(candle_index)` addition matches the standard **dispatch-table / interpreter (visitor) pattern**: one polymorphic call site per block type, eliminating the `block_type ==` ladders in `explanation.py` and the frontend.
- **#510 — promote the strategy validator to a shared, codegen-projected seam** (TP-ladder bounds, stop-loss range currently hand-duplicated in `ParameterForm.tsx`). The standard pattern is **Pydantic → JSON Schema → Zod codegen**: `model_json_schema()` output feeds a generator (e.g. `pydantic2zod`, or a JSON-Schema-to-Zod tool) that produces the TypeScript validation module, so the numeric bounds live once in `backend/app/services/strategy_validation.py`.
- **#511 — unify backtest submission (`submit(strategy, params) -> BacktestRun`)**, fixing the orphaned-`PENDING`-run risk when `session.commit()` succeeds but `queue.enqueue()` throws. The standard fix is the **transactional outbox pattern**: write the `BacktestRun` row and an outbox/enqueue-intent row in the same transaction; a relay (or the same request, retried) claims outbox rows with `SELECT ... FOR UPDATE SKIP LOCKED` and enqueues to RQ, deleting the row on success. This guarantees at-least-once enqueue (the RQ worker/job must stay idempotent, which `auto_update_strategies_daily`'s existing skip-if-already-running check already supports).
